# 削除の安全性の判断基準

コードを削除する前に、以下の「見えない参照」がないか必ず確認してください。

## 1. 動的なプロパティ参照（Dynamic Access）
リテラル文字列によるオブジェクトのプロパティ参照。
- 例: `obj['propertyName']`
- `grep_search` でプロパティ名自体を検索してください。

## 2. 動的なインポート（Dynamic Import）
- 例: `import(someVariable)`

## 3. 暗黙的な参照
- `package.json` の `scripts` や、`prisma.schema` などの設定ファイルで使用されているファイル名。
- Dockerfile や CI/CD パイプライン（`.github/workflows` など）からの参照。

## 4. テストコード
- プロダクションコードでは使われていないが、テストコード（`.spec.ts`, `.test.ts`, `test/` ディレクトリ）で重要、または参照されている場合。
- 機能を削除する場合は、対応するテストも同時に削除してください。

## 5. DTO や Entity
- バックエンドの API レスポンスとして定義されている DTO は、フロントエンド側でのみ使われている可能性があります（プロジェクトを跨いで検索してください）。

## 6. CSS の依存関係
- `.class-a` を削除した際、そのクラスを前提にした `+` や `>` などの CSS セレクタに影響がないか確認してください。

---

**迷った時は**: 完全に削除せず、一時的にコメントアウトするか、一度の PR で削除範囲を小さく保つことをお勧めします。
